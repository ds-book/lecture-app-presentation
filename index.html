Реактивные выражения reactive()

Для упрощения зависимостей данных можно использовать «реактивные проводники» (reactive conductors), которые находятся между источниками данных (например, input) и конечными точками (например, output).

Таким образом, reactive() зависит от различных вышестоящих входных данных и может использоваться для генерации выходных значений.

Используются аналогично функциям в обычном R-скрипте и помогают:

  • избегать повторения кода (принцип DRY)
  • разбивать сложные вычисления на более мелкие и модульные шаги
  • повышать вычислительную эффективность за счёт разделения и упрощения реактивных зависимостей

Советы по использованию reactive()

  • Код пишется аналогично функциям render*()

  • Если react_obj = reactive({...}), то любой потребитель должен обращаться к значению через react_obj(), а не react_obj

    • воспринимайте react_obj как функцию, которая возвращает текущее значение

    • Частая причина ошибки в R:

      ## Error: object of type 'closure' is not subsettable

  • Как и input, реактивные выражения могут использоваться только внутри реактивного контекста (например, render*(), reactive(), observe(), observeEvent() и т.д.)

    ## Error: Operation not allowed without an active reactive context. 
## (You tried to do something that can only be done from inside a reactive expression or observer.)

Элементы observer()

Создаются аналогично reactive(), однако наблюдатель (observer) не возвращает значения, поэтому используется исключительно ради своих побочных эффектов.

  • В большинстве случаев эти побочные эффекты заключаются в отправке данных в браузер клиента, например, для обновления элемента интерфейса.

  • Хотя это неочевидно из синтаксиса — результаты функций render*() на самом деле являются наблюдателями.

Пример с observer()

library(tidyverse)
library(shiny)
d = readr::read_csv("data/weather.csv")

d_vars = d |>
  select(where(is.numeric)) |>
  names()

shinyApp(
  ui = fluidPage(
    titlePanel("Weather Forecasts"),
    sidebarLayout(
      sidebarPanel(
        selectInput(
          "region", "Select a region",
          choices = sort(unique(d$region))
        ),
        selectInput(
          "city", "Select a city",
          choices = c(),
          multiple = TRUE
        ),
        selectInput(
          "var", "Select a variable",
          choices = d_vars, selected = "temp"
        )
      ),
      mainPanel( 
        plotOutput("plot")
      )
    )
  ),
  server = function(input, output, session) {
    
    d_city = reactive({
      req(input$city)
      d |>
        filter(city %in% input$city)
    })
    
    observe({
      cities = d |>
        filter(region == input$region) |>
        pull(city) |>
        unique() |>
        sort()
      
      updateSelectInput(
        inputId = "city", 
        choices = cities
      )
    })
    
    output$plot = renderPlot({
      d_city() |>
        ggplot(aes(x=time, y=.data[[input$var]], color=city)) +
        ggtitle(input$var) +
        geom_line()
    })
  }
)

Использование req()

Чтобы валидировать входные данные и запускать обработку только в случае их наличия, в реактивных сценариях можно использовать функцию req().

d_city = reactive({
  req(input$city)
  d |>
    filter(city %in% input$city)
})

Если значение отсутствует, дальнейшее выполнение реактивного кода (и кода зависимых потребителей) прекращается.

Элементы downloadButton() и downloadHandler()

Это UI- и серверные компоненты, необходимые для скачивания файла из вашего Shiny-приложения. Скачиваемый файл может быть любого типа и содержания.

downloadButton() — это частный случай actionButton() со специализированным синтаксисом на стороне сервера.

В частности, внутри определения server функция downloadHandler() привязывается к id кнопки через output, например:

output$download_btn <- downloadHandler(
  filename = function() {
    "data.csv"
  },
  content = function(file) {
    write.csv(data(), file, row.names = FALSE)
  }
)

Пример скачивания файла

Обработчик определяет:

  • функцию filename — для генерации имени файла по умолчанию

Модальные окна

Это всплывающие окна, которые позволяют отображать важные сообщения (например, предупреждения или ошибки) или другие UI-элементы, не загромождая основной интерфейс приложения.

Модальное окно состоит из набора Shiny UI-элементов (статических или динамических) и отображается только при активации (обычно с помощью кнопки actionButton или ссылки).

Они отличаются от других рассмотренных нами UI-элементов тем, что обычно определяются в серверной части приложения (server), а не в интерфейсе (ui).

)

Загрузка данных fileInput()

Этот виджет работает немного иначе, чем рассмотренные ранее: после загрузки файла он возвращает датафрейм с одной строкой на файл и следующими колонками:

  • name — исходное имя файла (из файловой системы клиента)
  • size — размер файла в байтах
  • type — MIME-тип файла, обычно определяемый по расширению
  • datapath — путь к временному файлу на сервере

Исходя из этого датафрейма, серверный код вашего приложения отвечает за фактический процесс чтения загруженного файла.

Советы по использованию fileInput()

  • input$upload по умолчанию равен NULL при загрузке приложения, поэтому хорошей практикой является использование req(input$upload) для зависимых реактивных выражений

  • Файлы в datapath являются временными, и к ним следует относиться как к эфемерным:

    • последующие загрузки могут привести к удалению предыдущих файлов

  • type — это лишь предположение — всегда валидируйте загруженные файлы и пишите защитный код

  • Аргумент accept помогает ограничить типы файлов, но не может предотвратить загрузку некорректных файлов

Пример с загрузкой файла

library(tidyverse)
library(shiny)

shinyApp(
  ui = fluidPage(
    titlePanel("Temperature Forecasts"),
    sidebarLayout(
      sidebarPanel(
        fileInput("upload", "Upload a file")
      ),
      mainPanel( 
        h3("Result"),
        tableOutput("result"),
        h3("Content:"),
        tableOutput("data")
      )
    )
  ),
  server = function(input, output, session) {
    output$result = renderTable({
      input$upload
    })
    
    output$data = renderTable({
      req(input$upload)
      ext = tools::file_ext(input$upload$datapath)
      validate(need(ext == "csv", "Please upload a csv file"))
      readr::read_csv(input$upload$datapath)
    })
  }
)

3️⃣ Публикация с использованием Github Actions

Автоматизация деплоя Shiny-приложений через GitHub Actions

GitHub Actions — встроенная платформа CI/CD для автоматизации рабочих процессов прямо в репозитории. Позволяет превратить ручную публикацию в одно нажатие git push.

Зачем использовать GitHub Actions для Shiny?

Преимущество Описание
Автоматизация Деплой запускается автоматически при пуше кода
Воспроизводимость Чистая среда Ubuntu каждый раз → нет конфликтов зависимостей
Версионирование История деплоев сохраняется в логах Actions
Безопасность Токены хранятся в Secrets, не попадают в код
Независимость от RStudio Работает даже если у вас локально R не установлен

Ключевые компоненты процесса

  1. Workflow-файл (.github/workflows/deploy.yml)
    YAML-сценарий, описывающий шаги сборки и деплоя.

  2. Пакет rsconnect
    Официальный клиент Posit для связи с shinyapps.io.

  3. Secrets & Variables (настройки репозитория)
    Токены доступа и имя приложения передаются безопасно.

  4. Зависимости (renv.lock или DESCRIPTION)
    Гарантируют, что на сервере установятся те же версии пакетов.

Как работает процесс (по шагам)

  1. Вы делаете git push в ветку main
  2. GitHub запускает виртуальную машину (Ubuntu)
  3. Устанавливается R, системные библиотеки и пакеты (renv)
  4. rsconnect формирует бандл и загружает его на shinyapps.io
  5. Сервер собирает приложение и запускает его
  6. Приложение доступно по ссылке: https://<account>.shinyapps.io/<app>/

Структура workflow-файла (минимальный пример)

name: Deploy to ShinyApps
on:
  push:
    branches: [main]
  workflow_dispatch: # Ручной запуск

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: r-lib/actions/setup-r@v2
        with: r-version: '4.4.0'
      - uses: r-lib/actions/setup-renv@v2
      - name: 🚀 Deploy
        run: |
          Rscript -e '
            install.packages("rsconnect")
            rsconnect::setAccountInfo(
              name = Sys.getenv("ACCOUNT"),
              token = Sys.getenv("TOKEN"),
              secret = Sys.getenv("SECRET")
            )
            rsconnect::deployApp(appName = Sys.getenv("APP_NAME"))
          '
        env:
          ACCOUNT: ${{ vars.SHINYAPPS_ACCOUNT }}
          TOKEN: ${{ secrets.SHINYAPPS_TOKEN }}
          SECRET: ${{ secrets.SHINYAPPS_SECRET }}
          APP_NAME: ${{ vars.SHINYAPPS_APP_NAME }}

Безопасность: Secrets vs Variables

Тип Что хранить Видно в логах? Где настраивать
secrets.* Токен, Secret, пароли ❌ Никогда Settings → Secrets
vars.* Имя аккаунта, имя приложения ✅ Да Settings → Variables

Золотое правило

Никогда не хардкодьте токены в YAML или R-скриптах! Используйте ${{ secrets.* }} и ${{ vars.* }}.

Чек-лист перед первым деплоем

  • Приложение стабильно работает локально (runApp())
  • Создан .Rbuildignore (исключает .github/, Dockerfile, renv/library/)
  • В GitHub настроены Variables: SHINYAPPS_ACCOUNT, SHINYAPPS_APP_NAME
  • В GitHub настроены Secrets: SHINYAPPS_TOKEN, SHINYAPPS_SECRET
  • Workflow-файл добавлен в .github/workflows/
  • Тестовый пуш выполнен → проверяем вкладку Actions

Совет для отладки

Добавьте workflow_dispatch: и запускайте вручную, пока не убедитесь, что всё работает. Это экономит время на лишних пушах.

4️⃣ Сборка и публикация Docker-образа

Контейнеризация Shiny-приложений с Docker

Docker — стандарт индустрии для упаковки приложений в изолированные, переносимые и воспроизводимые среды.
Позволяет забыть про фразу «у меня работает, а у тебя — нет».

Что такое Docker? Основные понятия

Термин Аналогия Описание
Image (Образ) Рецепт или ISO-файл Невозможный шаблон с ОС, R, пакетами и кодом приложения
Container (Контейнер) Запущенная программа Живой экземпляр образа, который выполняет код
Dockerfile Инструкция по сборке Текстовый файл с командами для создания образа
Docker Engine Двигатель Фоновая служба, которая управляет образами и контейнерами

💡 Контейнеры используют ядро хостовой ОС, поэтому они легче и быстрее виртуальных машин.

Зачем упаковывать Shiny в Docker?

Проблема без Docker Решение с Docker
❌ Зависимости конфликтуют на разных машинах ✅ Образ гарантирует одинаковую среду всегда
❌ Долгая настройка сервера (R, пакеты, sys-libs) ✅ Готовый образ запускается за секунды
❌ Сложный деплой на продакшен ✅ Один образ → любой хостинг/облако
❌ «Работает локально, падает на сервере» ✅ Идентичная среда на всех этапах разработки

Для Shiny это особенно важно: гео-пакеты (sf, terra), системные библиотеки и версии R часто вызывают ошибки при переносе. Docker решает это раз и навсегда.

Dockerfile — рецепт сборки образа

# Базовый образ с R и Shiny
FROM rocker/shiny:4.4.0

# Системные зависимости для пакетов
RUN apt-get update && apt-get install -y \
    libcurl4-openssl-dev libssl-dev libxml2-dev && \
    rm -rf /var/lib/apt/lists/*

# Копируем зависимости и устанавливаем их
COPY renv.lock .
RUN Rscript -e 'install.packages("renv"); renv::restore()'

# Копируем код приложения
COPY . /app
WORKDIR /app

# Открываем порт Shiny
EXPOSE 3838

# Команда запуска
CMD ["R", "-e", "shiny::runApp('/app', host='0.0.0.0', port=3838)"]

.dockerignore — исключает renv/library/, .git/, *.RData из сборки, ускоряя процесс и уменьшая размер образа.

Что такое Registry?

Registry (Реестр) — хранилище Docker-образов, аналог GitHub для кода.

Действие Команда Назначение
Загрузить образ docker pull <image> Скачать готовый образ из реестра
Отправить образ docker push <image> Опубликовать свой образ в реестре
Популярные реестры Docker Hub, GitHub Container Registry (GHCR), AWS ECR Публичные или приватные хранилища

В CI/CD pipeline образ собирается → тестируется → пушится в реестр → сервер автоматически pull-ит и запускает новую версию.

https://docs.docker.com/get-started/introduction/build-and-push-first-image/

Docker: image, container, registry

Как запускать контейнеры?

# 1. Сборка образа из Dockerfile
docker build -t my-shiny-app:latest .

# 2. Запуск в фоне с пробросом порта
docker run -d \
  --name shiny-prod \
  -p 3838:3838 \ # связывает порт хоста с портом контейнера.
  -e API_KEY=secret123 \ # передаёт переменные окружения
  my-shiny-app:latest

# 3. Проверка работы
docker ps          # список запущенных контейнеров
docker logs shiny-prod  # логи приложения

# 4. Остановка и удаление
docker stop shiny-prod
docker rm shiny-prod

Типичный workflow разработки

  1. Пишете код → фиксируете зависимости (renv::snapshot())
  2. Собираете образ: docker build -t my-app .
  3. Тестируете локально: docker run -p 3838:3838 my-app
  4. Пушите в registry: docker tag my-app ghcr.io/user/my-app && docker push ...
  5. Разворачиваете на сервере или в GitHub Actions
  6. При обновлениях повторяете шаги 1–5

Используйте многоэтапные сборки (multi-stage builds) для продакшена, чтобы отделить среду компиляции от лёгкого финального образа.

Чек-лист перед контейнеризацией Shiny

  • Приложение стабильно работает локально
  • Создан Dockerfile на базе rocker/shiny или rocker/verse
  • Добавлен .dockerignore (исключает renv/library/, .git/, временные файлы)
  • Порт 3838 проброшен в EXPOSE и docker run -p
  • Секреты передаются через -e VAR=value или .env, а не зашиты в код
  • Образ протестирован локально перед пушем в registry

Не храните .RData, .Rhistory или файлы с персональными данными в образе! Используйте внешние хранилища или загрузку через fileInput().

Команды для сборки и публикации образа

# Логинимся в DockerHub
docker login

# Собираем образ для репозитория в DockerHub
docker build -t rymbln/demo-shiny-app .

# Отправляем собранный образ на DockerHub
docker push rymbln/demo-shiny-app

# Запускаем образ из DockerHub
docker run -d --rm --name my-demo-shiny-app -p 3838:3838 rymbln/demo-shiny-app
